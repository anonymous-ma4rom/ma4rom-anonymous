## 一、投稿前必须修正的问题

### 1. No context 的运行时间与实验文件不一致

论文 Table 4 使用：

```text
No context runtime = 1,807.89 s
```

但实验文件 `output/ablation/context_efficiency/cmt_structured_context_efficiency.csv` 中记录的是：

```text
No context runtime = 4,407.89 s
failed_attempts = 15
```

不能在没有新实验依据的情况下把 `4,407.89` 改成 `1,807.89`。可选处理方式只有两种：

1. 重新运行 No context，得到受控且可复现的新时间，再更新表格。
2. 保留原始时间，但明确说明该时间受 15 次 API 失败和回退影响，因此效率结论只比较 Full context 与 Confidence controlled。

这是结果可信性问题，优先级最高。

### 2. “去除全部外键后所有场景仍超过 LLM4VKG”不成立

论文 Table 5 的 FK-removed `Ren.` 为 `0.8851`。Table 2 中 LLM4VKG 的 Adjusted Naming 平均值为：

```text
(0.86 + 0.92 + 0.93) / 3 = 0.9033
```

因此 `0.8851 < 0.9033`，不能写：

> Even after all declared foreign keys are removed, MA4ROM outperforms LLM4VKG across all evaluated scenarios.

应改成更弱且真实的结论，例如“在多数场景中仍超过 LLM4VKG”，并明确 Rename 是例外；或者重新确认 Table 5 的聚合口径。

### 3. Component Ablation 的 Full System 与主结果不一致

Table 3 的完整系统结果与 Table 2/Table 5 不一致：

| Scenario | Table 3 | Table 2/Table 5 |
|---|---:|---:|
| Rename | 0.9597 | 0.9367 |
| Restr. | 0.8608 | 0.8693 |
| Mixed | 0.8269 | 0.9649 |
| NoFK | 0.8205 | 0.8462 |

### 4. 4.2 伪代码与真实代码流程不一致

论文 Algorithm 1 表示：候选达到 medium 阈值时直接接受 top-1，否则进入上下文增强。

实际代码 `DPMapping/data_property_mapping_agent.py` 的流程是：

- high confidence：直接接受 top-1；
- medium/low confidence：先调用 LLM 在候选中选择；
- 只有最终被收集为 low confidence 的表/列才进入上下文增强。

因此论文缺少“初始 LLM selection”步骤，并错误地把 medium confidence 写成直接接受 top-1。需要选择一种真实口径：

1. 按代码修改 Algorithm 1：high 直接接受，其他候选先经 LLM selection，再将 low confidence 结果送入 context enhancement。
2. 按论文修改代码并重新跑全部实验。

不能保留当前不一致状态。

### 5. 4.3 对 OP 候选生成的描述不完整

论文写成“仅保留 domain/range 与精化类兼容的 OP”，但代码 `OPMapping/equivalence_op_module.py` 还会使用：

- ontology restrictions；
- SQL construction matches；
- table/column/constraint name pattern matches；
- normal/reversed direction for relation tables；
- inverse/subproperty information供 LLM 判断。

老师也明确要求利用 restriction 和 construction。当前 Figure 3 画出了 `Restriction and Construction Matching`，正文和规则表却完全没有解释，形成图、文、代码三者不一致。

4.3 至少要用一句话和一个形式条件说明 restriction/construction 如何参与候选生成，或者从 Figure 3 删除该组件并按简化方法重跑/确认实现。

### 6. “First Order Logic Rules”表述过强

Rule 1：

```text
C* = {C} ∪ Ancestor(C) ∪ Children(C)
```

是集合定义，不是一阶逻辑规则。Rule 2 和 Rule 3 才是一阶逻辑风格的蕴含式。因此标题 `First Order Logic Rules Based Object Property Refinement` 不严格。

建议统一称为：

```text
Object Property Mapping with Logic Rules
```

同时，代码使用传递闭包，可匹配所有祖先和后代类；论文只写 `Children(C)`，范围过窄。应使用 `Descendants(C)`，或者明确 `Children` 表示传递后代闭包。

## 三、方法与实现不一致

### 1. 4.2 对 type column 的表述过宽

全文使用 `enumerated columns`，Algorithm 1 使用 `IsEnumerated(a)`。这会表达成“所有枚举列都映射到 ontology subclasses”，并不符合代码。

代码处理的是承担 subclass discriminator 作用的 type columns。建议统一为：

```text
type-column values
subclass-discriminator columns
IsTypeColumn(a)
```

如果保留 `enumerated`，至少应定义为：

```text
an enumerated type column used as a subclass discriminator
```

### 2. 4.2 省略了 type-column mapping 的规则优先流程

`RealValue/real_value_enhancement_agent.py` 中的实际流程包含：

- 低基数/type-column 判定；
- 根据值形式、类候选、子类层次和 FK 覆盖差异进行规则打分；
- 高置信规则结果直接锁定；
- 未解决的值再交给 LLM。

论文当前只描述“按值分组后让 LLM 选择 subclass”，不足以复现实验方法。至少需要在 4.2.3 增加一句：

```text
The value groups are first evaluated by rule-based subclass
ranking, and unresolved values are then passed to the LLM.
```

### 3. 4.3 对 OP 候选生成的描述明显少于实际实现

论文当前写成：类精化后，仅通过 domain/range 生成 OP candidates。

当前 `OPMapping/equivalence_op_module.py` 实际还使用：

- equivalence/inclusion column matching；
- ontology restrictions；
- SQL construction annotations；
- table、column 和 constraint name patterns；
- relation-table normal/reversed directions；
- inverse and subproperty metadata。

而且代码的候选保留条件是 endpoint、restriction、construction 或 name pattern 任一命中，并非论文所写的“只保留 domain/range compatible properties”。

老师要求等价列不是主要亮点，可以只写一句，但不能完全不写。4.3 应至少说明：IND/等价列匹配用于确定关系列及方向，ontology domain/range、restriction 和 construction 用于生成候选，LLM 再从候选中选择。

### 4. `Class refinement` 目前被写成了逻辑推导，但实际是候选扩展

Table 1 的：

```text
C* = {C} ∪ Ancestor(C) ∪ Children(C)
```

并不能从 `Person` 逻辑推出某个实例一定是 `Author` 或 `Reviewer`。它只是把 ontology hierarchy 中的类加入候选空间，之后再由关系信息和 LLM 选择。

因此应称为：

```text
class candidate refinement
class candidate expansion using the ontology hierarchy
```

而不是暗示 ontology axioms 已经完成确定性的 subclass inference。

另外，代码允许传递祖先和传递后代，论文的 `Children(C)` 只表示直接子类。应改为 `Descendants(C)`，或明确 Children 表示传递闭包。

### 5. Rule 1 仍不是严格的一阶逻辑规则

Rule 1 是集合闭包定义，Rule 2 和 Rule 3 才是蕴含式规则。当前标题 `Object Property Mapping Based on Logic Rules` 可以保留，但不要再在摘要或正文中声称三个式子全部是 first-order logic rules。
