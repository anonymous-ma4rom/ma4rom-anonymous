## 4. Preliminaries 怎么改

老师的意思是：先给数据库模式和本体元素集合，再定义总映射函数。

不要把 `C_enum` 直接塞进数据库模式元组里，因为它不是数据库教科书里的 schema 原生组成部分，而是从列和值中抽出来的映射对象。

推荐结构：

1. 定义数据库：

```latex
Let \(D=(D_S,I)\) be a relational database, where \(D_S\) is the database schema and \(I\) is the database instance. Each table \(t\in T\) is represented by its attributes \(A(t)\), key attributes \(A_k(t)\), and FK attributes \(A_f(t)\).
```

2. 定义本体元素：

```latex
Let \(O_N=(E_c,E_{\mathrm{dp}},E_{\mathrm{op}})\) be the ontology element set, where \(E_c\), \(E_{\mathrm{dp}}\), and \(E_{\mathrm{op}}\) denote classes, datatype properties, and object properties.
```

3. 定义总映射：

```latex
\[
f:D_S\rightarrow O_N
\]
```

4. 定义三个子映射：

```latex
\[
f_C:T\cup C_{\mathrm{enum}}\rightarrow E_c
\]
```

其中 \(C_{\mathrm{enum}}\) 不是数据库 schema 原生集合，而是从包含枚举值的列中抽取的条件集合，例如 \(\mathrm{Paper.TYPE}=1\)。

```latex
\[
f_D:C_{\mathrm{attr}}\rightarrow E_{\mathrm{dp}}
\]
```

其中 \(C_{\mathrm{attr}}\) 是需要映射到 datatype property 的属性集合。

```latex
\[
f_O:T_r\cup C_f\rightarrow E_{\mathrm{op}}
\]
```

其中 \(T_r\) 是 relationship tables，\(C_f\) 是 FK columns。

需要补一个 FK 引用函数供 4.3 使用：

```latex
\[
f_k:C_f\rightarrow T
\]
```

其中 \(f_k(a_f)\) 返回 FK column \(a_f\) 引用的目标表。不要再写 `FK(t)`。

## 5. 4.2 修改方向


### 4.2.1 Overall Algorithm

算法要简化，不要每一行都像程序一样写。重点保留：

1. 先处理普通 datatype property mapping。
2. 对候选做检索、匹配、排序。
3. 如果 top1 分数高且 margin 足够大，直接接受。
4. 否则采样真实值并连接相关表，交给 LLM 重新选择。
5. 再处理 enumerated values，按值分组并连接相关表，映射到 ontology subclasses。

不要在算法里把 `Retriever`、`LLMMatcher`、`RankCandidates`、`TopTwo` 写得太细。可以抽成一个函数，例如：

```latex
\State \(L_a\gets \mathrm{RetrieveAndRank}(a,E_{\mathrm{dp}})\)
```

然后在 4.2.2 解释这个函数如何结合 Retriever、Matcher、score 和 margin。

### 4.2.2 Ontology Retrieval and Confidence Estimation

这里一段写检索，一段写匹配和置信度。不要展开成像 LLM4VKG 那样晦涩，但可以借鉴它的函数风格。

必须保留：

```latex
\[
\mathrm{margin}(L)=s_1-s_2
\]
```

并说明 \(s_1\)、\(s_2\) 是 top1 和 top2 的 matching score。高置信条件：

```latex
\[
d(L)=
\begin{cases}
\mathrm{High}, & s_1\geq\theta_h \land s_1-s_2\geq\Delta_h,\\
\mathrm{Low}, & \mathrm{otherwise}.
\end{cases}
\]
```

其中 `High` 直接接受 top1，`Low` 进入 context enhancement。

这里是 4.2 的重点。建议用 \(\Gamma\) 表示增强上下文，避免和本体元素 \(E_c,E_{\mathrm{dp}},E_{\mathrm{op}}\) 冲突。

### 4.3.1 IND Based FK Discovery

保留 IND 公式即可，不要写太长：

```latex
\[
\mathrm{IND}(s,t)=\frac{|V_s\cap V_t|}{|V_s|}
\]
```

如果 \(\mathrm{IND}(s,t)\geq\theta_{\mathrm{IND}}\)，则 \(s\) 被认为是引用 \(t\) 的 candidate FK。这里统一写 `FK`，不要再写完整 foreign key。

### 4.3.2 Logic Rule Based OP Candidate Retrieval

不要再叫 `Axioms`。表格 caption 建议：

```latex
\caption{Logic rules for object property candidate retrieval.}
```

表格三列：

```latex
No. | Rule | Example
```

不要写很宽的 Name 列。No. 列窄一点，Rule 居中或左对齐，Example 用 Figure 的 `Paper.R2`、`R1` 例子。

核心逻辑：

规则 ①：domain and range constraint。

```latex
\[
p(x',y')\Rightarrow x'\in dom(p)\land y'\in ran(p)
\]
```

规则 ②：retrieval condition。

FK column 情况：

```latex
\[
dom(p)\subseteq f_C(t),\qquad ran(p)\subseteq f_C(f_k(a_f))
\]
```

Relationship table 情况：

```latex
\[
(dom(p)\subseteq f_C(t_1)\land ran(p)\subseteq f_C(t_2))
\lor
(dom(p)\subseteq f_C(t_2)\land ran(p)\subseteq f_C(t_1))
\]
```

推理结果 ③：FK column OP mapping。

```latex
\[
p(x',y')\Rightarrow x'\in f_C(t)\land y'\in f_C(f_k(a_f))
\]
```

推理结果 ④：relationship table OP mapping。

```latex
\[
p(x',y')\Rightarrow
(x'\in f_C(t_1)\land y'\in f_C(t_2))
\lor
(x'\in f_C(t_2)\land y'\in f_C(t_1))
\]
```

正文解释方式：

```text
Rule ① follows the domain and range semantics of ontology object properties. Rule ② is the retrieval condition derived from the mapped database endpoints. For an FK column, the source table and the referenced table provide the two endpoint classes. For a relationship table, the two referenced tables provide the endpoint classes, and both directions are checked because the table name may not indicate the represented relationship. Rules ③ and ④ are the inferred endpoint typing results used to retrieve OP candidates.
```

不要在表格里写 `p\in P_{\mathrm{cand}}(t)`。如果需要候选集合，放正文一句：

```latex
\[
P_{\mathrm{cand}}(r)=\{p\in E_{\mathrm{op}}\mid p \text{ satisfies the corresponding retrieval condition}\}.
\]
```

### 4.3.3 LLM Based Pattern Context OP Selection

这里写 LLM 如何选：

1. 输入：候选 OP 集合、relationship table 或 FK column、两端表的 class mapping、schema pattern、instance values。
2. 输出：top1 OP。
3. 说明 LLM 不自由生成 OP，只能从规则检索出的候选中选。
