# 老师讨论后的论文重构方向

## 一、这次修改的中心思想

本轮不是继续修饰原有表述，而是重新建立全文的形式化主线：

1. **Preliminaries 定义数据库元素到本体元素的映射关系，而不是只定义系统输入输出。**
2. **Section 4.2 说明如何检索、匹配并利用上下文完成 class 和 datatype property mapping。**
3. **Section 4.3 分开描述数据库层的关系发现与本体层的候选推理。**
4. **对象属性候选生成的结果是一个集合，LLM 最后从集合中选择 top-1。**
5. **数据库关系、映射断言和本体公理必须分层描述，不能混成同一种“规则”。**

### 2. 必须修正的术语

#### Semantic Loss

引用的 CTA/CEA 工作使用的是：

> semantic loss of column names or cell values

因此不能写 `Semantic Loss of Column`。建议统一为：

```text
semantic loss of column names
```

它只用于描述 DP mapping 的问题。关系表或外键列名称的问题没有现成的 `Semantic Loss of Relationship` 原文支持，建议写：

```text
relationship names that provide insufficient semantic information
```

或者：

```text
semantic loss in relationship names
```

后一种属于本文自己的概括，必须先定义后使用。

#### Enumerated Values

不能写：

```text
mapping enumerated value values
column contains enumerated value
```

建议统一为：

```text
mapping enumerated values to ontology subclasses
columns containing enumerated values
```

为了满足老师“不区分枚举列和枚举值术语”的要求，正文主要谈 `enumerated values`。需要指代所在列时写：

```text
a column containing enumerated values
```

映射对象仍然是条件 \(a=v\)，不是整列：

\[
\sigma_{a=v}(T)\longmapsto C_v.
\]

#### Confidence

LLM4VKG 中：

- `Retriever` 检索候选本体元素；
- `Matcher` 返回匹配元素以及 High/Medium/Low matching degree；
- `Namer` 在匹配不足时生成新的本体术语。

`Namer` 不是置信度判断模块。MA4ROM 使用固定目标本体，不生成新类或属性，因此不应把 Namer 写进自己的流程。

当前实现根据 top-1 分数 \(s_1\) 和 top-1/top-2 分差 \(s_1-s_2\) 决定是否启用上下文增强。建议称为：

```text
score and margin based confidence assessment
```

或者在不想增加术语时直接写：

```text
a confidence criterion based on the top-ranked score and the score margin
```

不要声称这是 calibrated confidence probability。

#### Foreign Key Discovery

IND 论文研究的是 `inclusion dependency discovery`，并没有直接把所有 IND 称为 foreign key。

Mapping-pattern 文献明确指出：外键是满足被引用属性构成 key 的一类 inclusion dependency。因此本文应写：

```text
IND-based candidate foreign key discovery
```

而不是直接声称 IND 能确定真实外键。推荐过程：

1. discover INDs；
2. 检查目标列是否为 primary/unique key；
3. 将满足阈值和 key 条件的列对作为 candidate foreign keys；
4. 交给后续对象属性映射处理。

Section 4.3 正文首次出现时定义：

```text
inclusion dependency (IND)
```

此后可在标题中使用：

```text
IND-Based Candidate Foreign Key Discovery
```

## 三、Preliminaries 必须重写为真正的映射定义

原来的

\[
F_{\mathrm{R2O}}:(D,O)\rightarrow M
\]

只是系统输入输出，不足以定义数据库元素如何映射到本体元素。

按照 VKG mapping 文献，映射断言应写成 source query 到 target atom：

\[
m:\quad s:Q(\mathbf{x})\quad\longrightarrow\quad
t:L(\mathbf{t}(\mathbf{x})),
\]

其中：

- \(Q(\mathbf{x})\) 是数据库上的 source query；
- \(L\) 是本体中的 class、datatype property 或 object property；
- \(\mathbf{t}(\mathbf{x})\) 是根据数据库列生成 RDF 个体或字面值的模板。

### 1. Class Mapping

表到类：

\[
m_C:\quad Q_T(\mathbf{k})\longrightarrow C_T(t_T(\mathbf{k})).
\]

包含枚举值的条件到子类：

\[
m_{C_v}:\quad \sigma_{a=v}(T)\longrightarrow
C_v(t_T(\mathbf{k})).
\]

这里必须补上普通的 table-to-class mapping，不能只写 subclass mapping。

### 2. Datatype Property Mapping

\[
m_D:\quad Q_{T,a}(\mathbf{k},a)\longrightarrow
d(t_T(\mathbf{k}),a),
\]

其中 \(d\) 是与列 \(a\) 对应的 datatype property。

### 3. Object Property Mapping

\[
m_O:\quad Q_r(\mathbf{k}_s,\mathbf{k}_t)\longrightarrow
p(t_s(\mathbf{k}_s),t_t(\mathbf{k}_t)),
\]

其中 \(Q_r\) 可以来自：

- entity table 中的 foreign key column；
- 连接两个实体表的 relationship table。

这三个定义才真正表达“数据库的什么映射到本体的什么”。

## 四、Section 4.2 的重构方向

### 建议标题

```text
4.2 Class and Datatype Property Mapping with Context Retrieval
```

这样比堆叠多个带连字符的定语更清楚。

### 4.2.1 Overall Algorithm

只介绍总体流程：

1. 先处理普通 datatype property mapping；
2. Retriever 从本体检索候选；
3. Matcher 对候选打分并排序；
4. 根据 top-ranked score 和 score margin 判断 high/low confidence；
5. high-confidence mapping 直接接受；
6. low-confidence mapping 使用 instance values 和 schema information 增强；
7. 再处理 enumerated values 到 ontology subclasses 的映射。

不要在正文逐行解释算法函数。

### 4.2.2 Ontology Retrieval and Matching

分成两段：

**Retrieval.** 根据列名、表类、本体局部名称、domain 等信息，从目标本体中检索有限候选集合。

**Matching.** 对检索候选进行打分与排序。使用：

\[
c_{\max}(L)=s_1,\qquad
c_{\mathrm{margin}}(L)=s_1-s_2.
\]

满足两个阈值时为 high-confidence，否则为 low-confidence，并进入上下文增强。

不使用 LLM4VKG 的 Namer，因为本文不生成新的本体术语。

### 4.2.3 Context Enhancement for Class and Datatype Property Mapping

明确 context 的两个来源：

1. **instance values**：采样单元格和真实元组；
2. **schema information**：表名、列名、PK/FK 和相关表结构。

多表连接得到的是真实 relational context，可写成：

```text
instance values obtained through foreign-key joins, together with schema information
```

本节分别说明：

- DP：上下文帮助判断 VALUE 对应 title；
- Class：按 \(a=v\) 分组后，上下文帮助判断枚举值对应哪个 subclass。

## 五、Section 4.3 的重构方向

### 建议标题

```text
4.3 Object Property Mapping with Ontology Reasoning
```

如果还要强调模式：

```text
4.3 Pattern-Guided Object Property Mapping with Ontology Reasoning
```

但必须先确保 pattern 在方法中确实参与 OP 分支决策。

### 总体思想必须分成两个层次

#### 数据库层

从显式 FK 或 IND 中确定可能连接哪些表：

\[
c_s\rightarrow k_t.
\]

这一层只发现数据库关系，不决定 object property。

#### 本体层

利用表的 class mappings、class hierarchy、object-property domain/range 以及 subproperty axioms，生成 object property candidate set：

\[
\mathcal{P}_{cand}(r)\subseteq\mathcal{P}_o.
\]

最后 LLM 使用 schema/context 从集合中选择：

\[
p^{*}=\operatorname*{arg\,max}_{p\in\mathcal{P}_{cand}(r)}
\mathrm{LLMScore}(p\mid r,\mathrm{context}).
\]

### 4.3.1 IND-Based Candidate Foreign Key Discovery

内容保持短：

1. 定义 IND；
2. 写 inclusion ratio；
3. 目标列必须满足 key 条件；
4. 输出 candidate foreign keys；
5. 解释为什么小规模 RODI 数据库采用确定性 profiling 方法，而不是需要训练数据的 ML 方法。

不要声称 IND discovery 论文直接提出了 foreign key discovery。准确说法是：

> We use discovered INDs to identify candidate foreign keys.

### 4.3.2 Object Property Candidate Generation with Ontology Reasoning

原标题 `Logic Rules Based Object Property Refinement` 不准确，建议改为：

```text
Object Property Candidate Generation with Ontology Reasoning
```

因为该步骤的输出是候选集合，而不是单个 assignedTo。

## 六、本体公理与推理应如何写

### 1. 公理不是“输入输出函数”

OWL 2 公理是被声明为真的领域陈述，例如：

\[
Author\sqsubseteq Person
\]

\[
hasAuthor\sqsubseteq relatedTo
\]

\[
\exists hasAuthor.\top\sqsubseteq Paper
\]

\[
\exists hasAuthor^{-}.\top\sqsubseteq Author.
\]

它们分别表示：

- subclass axiom；
- subobject-property axiom；
- object-property domain axiom；
- object-property range axiom。

### 2. 推理与候选生成必须区分

由

\[
Author\sqsubseteq Person,\qquad Author(y)
\]

可以推出：

\[
Person(y).
\]

但不能由 \(Person(y)\) 推出 \(Author(y)\)。因此：

```text
Person -> Author/Reviewer
```

不是 OWL 逻辑蕴含，而是候选类精化：

\[
\mathrm{Refine}(Person)=
\{Person,Author,Reviewer,\ldots\}.
\]

最终具体选择 Author 还是 Reviewer，需要 domain/range、relationship context 和 LLM 判断。

### 3. Domain 和 Range 的正确含义

如果本体包含：

\[
\exists assignedTo.\top\sqsubseteq Paper,
\qquad
\exists assignedTo^{-}.\top\sqsubseteq Reviewer,
\]

并且已经有对象属性断言：

\[
assignedTo(x,y),
\]

那么可以推出：

\[
Paper(x),\qquad Reviewer(y).
\]

反过来，仅知道 \(Paper(x)\) 和 \(Reviewer(y)\)，不能在逻辑上推出
\(assignedTo(x,y)\)。本文只能利用 domain/range 过滤候选，不能声称它们推导出了关系。

## 七、assignedTo 例子应按步骤解释

### Step 1: 数据库关系发现

relationship table \(R1(Paper,Reviewer)\) 的两列分别引用：

\[
R1.Paper\rightarrow Paper.ID,
\qquad
R1.Reviewer\rightarrow Person.ID.
\]

如果 FK 缺失，4.3.1 使用 IND 得到 candidate foreign keys。

### Step 2: 获取已有 class mappings

\[
f_C(Paper)=cmt:Paper,
\qquad
f_C(Person)=cmt:Person.
\]

### Step 3: 类候选精化

根据 subclass axioms：

\[
cmt:Author\sqsubseteq cmt:Person,
\qquad
cmt:Reviewer\sqsubseteq cmt:Person.
\]

构造：

\[
\mathrm{Refine}(cmt:Person)=
\{cmt:Person,cmt:Author,cmt:Reviewer\}.
\]

这是候选扩展，不是从 Person 逻辑推出 Reviewer。

### Step 4: 根据 domain/range 生成 OP 候选集合

假设本体中：

\[
Dom(cmt:assignedTo)=cmt:Paper,
\quad
Ran(cmt:assignedTo)=cmt:Reviewer,
\]

\[
Dom(cmt:hasAuthor)=cmt:Paper,
\quad
Ran(cmt:hasAuthor)=cmt:Author.
\]

则：

\[
\mathcal{P}_{cand}(R1)
=\{cmt:assignedTo,cmt:hasAuthor,\ldots\}.
\]

此处结果必须是集合，不能直接写成 assignedTo。

### Step 5: LLM constrained selection

LLM 根据 relationship-table schema、列名和实例上下文，从候选集合中选择：

\[
p^{*}=cmt:assignedTo.
\]

### Step 6: 生成 mapping assertion

\[
\begin{aligned}
s:\;&\texttt{SELECT Paper, Reviewer FROM R1}\\
t:\;&cmt:assignedTo(t_{Paper}(Paper),t_{Person}(Reviewer)).
\end{aligned}
\]

这样例子完整展示了：

1. 数据库关系发现；
2. 数据库到本体的 class mappings；
3. subclass candidate refinement；
4. domain/range candidate filtering；
5. LLM top-1 selection；
6. 最终 R2RML mapping generation。
